## Notes
**Purpose of `@NgModule`**
- Centralizes the configuration of relationships between UI elements (components, directives, pipes) and data providers (services, guards).
- Replaces early Angular 2 patterns that required individual dependency declarations across every component and service file.

**Metadata Configuration & Visibility**
- `declarations`: Registers components, directives, and pipes. Items are module-scoped (private) by default.
- `exports`: Exposes declared items to external modules. Functions analogously to public visibility in object-oriented programming.
- `imports`: Allows a module to consume components, directives, or pipes that another module has exported. Together, `declarations`, `imports`, and `exports` govern cross-module sharing.
- `bootstrap`: Specifies the root component that initializes the application. Defaults to `AppComponent` (auto-generated by Angular CLI).
- `providers`: Registers injectable services and guards. Registered providers are accessible to all components within the module and to any module that imports it.

**Application Organization & Refactoring**
- Default CLI projects place all components and services in a single root `AppModule`, which becomes cluttered and difficult to maintain as the application scales.
- Recommended practice: Extract components and services into dedicated feature modules while retaining only the bootstrapped component in the root module.

**Routing Module Architecture**
- Must contain zero `declarations`.
- May include `providers` for route guards.
- Must always export `RouterModule`.
- Enables imperative component loading upon route activation. Complex applications should utilize multiple routing modules to improve organization and enable lazy loading.

## Key Takeaways
- `@NgModule` acts as a centralized dependency and visibility manager, eliminating fragmented configuration across individual files.
- Module visibility is explicit: items are private by default and only become accessible externally when added to `exports` and consumed via `imports`.
- Scalable Angular applications require decomposing monolithic root modules into focused feature modules to maintain code clarity and reduce technical debt.
- Routing modules follow a strict configuration contract (no declarations, optional guard providers, mandatory `RouterModule` export) and serve as the foundation for modular routing and lazy loading.

## Notes
**Feature Modules**
- Purpose: Encapsulate business logic and components tied to a specific application domain (e.g., a contacts section).
- Composition: Primarily contain `declarations` and may include `providers`. They should never include `exports`.
- Dependencies: Require `CommonModule` to access standard Angular structural directives (e.g., `*ngIf`, `*ngFor`).
- Implementation: Generated via Angular CLI. Feature-specific files are migrated into the new module, which is subsequently imported into the root module. This pattern significantly reduces root module clutter, especially when managing dozens of related components.

**Shared (Widget) Modules**
- Purpose: Distribute reusable UI components across multiple parts of the application.
- Composition: Structurally identical to feature modules but explicitly map `declarations` to the `exports` array.
- Use Cases: Common interface elements such as social media icons, loading spinners, and other reusable widgets.
- Implementation: Components are registered in both `declarations` and `exports`. Importing the shared module into the root or feature modules grants global access to its exported components.

**Core (Service) Modules**
- Purpose: Centralize application-wide services for dependency injection.
- Composition: Contain zero `declarations`; focus exclusively on `providers`.
- Use Cases: Global utilities such as authentication handlers, toast notification services, or data-fetching providers.
- Implementation: Generated via CLI and populated with service files. `CommonModule` is omitted since the module does not handle templates. The module is imported into the root to register global providers.

**Root Module Optimization**
- After extracting feature, shared, and core modules, the root module (`AppModule`) maintains a minimal footprint.
- The refactored root typically retains only the bootstrap component, essential declarations, and a consolidated `imports` array.
- All `providers` are offloaded to dedicated modules, resulting in a cleaner, more maintainable architecture.

## Key Takeaways
- Feature modules isolate domain-specific logic and must not export components.
- Shared modules enable cross-module reuse by exporting declared UI widgets.
- Core modules centralize global services and dependency injection without component declarations.
- Strategic module separation drastically reduces root module complexity and improves application scalability.
